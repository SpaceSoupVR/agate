================================================================================
 AGATE — IMMEDIATE-MODE UI TOOLKIT (API REFERENCE)
================================================================================

agate is a small immediate-mode UI layer built on top of space_soup's ui2d
primitives (Area/Item/Shape/Text/Font/Color/Align). It provides a themeable
widget set (Ui), a multi-line text editor widget (TextEditor), and the input
plumbing (UiInput/MouseButton/NamedKey) that drives both. Nothing here talks
to wgpu directly — every widget just appends (Area, Item) pairs to a list
that the host app hands to space_soup::ui2d::Overlay or a WorldPanel.

Crate layout:
  src/
    lib.rs              – module wiring + public re-exports
    input.rs            – per-frame input snapshot (UiInput) + key/button enums
    theme.rs             – color palette, spacing/size constants, Theme
    text_editor/          – the TextEditor widget (multi-line buffer + JSON highlight)
    widgets/                – the Ui struct and every widget method

--------------------------------------------------------------------------------
 lib.rs — Crate root / public API surface
--------------------------------------------------------------------------------
Declares the four top-level modules (theme, text_editor, input, widgets) and
re-exports the names host code actually imports:
  - `Theme` (from theme)
  - `TextEditor` (from text_editor)
  - `UiInput`, `MouseButton` re-exported as `AMouseButton` (from input) — the
    "A" prefix exists purely to avoid colliding with winit's own MouseButton
    in the host app.
  - `Ui`, `WidgetId` (from widgets)
  - `Area`, `Item`, `Color`, `Font`, `Align` re-exported from
    `space_soup::ui2d`, so callers building `(Area, Item)` lists (e.g. a
    custom widget, or PlacedObject inspector code) don't need a second
    `use space_soup::ui2d::...` line alongside `use agate::...`.

This file has no logic — it's purely the module tree and what's public.

--------------------------------------------------------------------------------
 input.rs — Per-frame input snapshot
--------------------------------------------------------------------------------
The host app (the windowing/event-loop code) builds one `UiInput` per frame
and hands it to `Ui::begin_frame`. Everything else in agate reacts to this
snapshot rather than touching the OS event loop directly.

  - `MouseButton` — `Left | Right | Middle`. Only `Left` is currently read
    anywhere in `widgets/`, but all three are tracked in case a widget needs
    right-click (e.g. a future context menu) later.
  - `UiInput` — one frame's worth of input:
      - `mouse_pos: (f32, f32)` — current cursor position in window pixels.
      - `mouse_held` / `mouse_pressed` / `mouse_released: Vec<MouseButton>` —
        which buttons are currently down, transitioned down this frame, or
        transitioned up this frame, respectively. Vecs (not bitflags) so the
        host can push from an arbitrary winit event stream without needing
        to know agate's button set in advance.
      - `scroll_y: f32` — accumulated scroll delta this frame.
      - `text: String` — committed text input this frame (already IME/dead-key
        resolved by the host) for `text_input`/`text_editor` to insert.
      - `keys: Vec<NamedKey>` — non-printable navigation/editing keys
        pressed this frame (arrows, Home/End, Backspace, etc).
      - `cmd` / `shift` / `alt: bool` — modifier state.
      - `dt: f32` — frame delta time, used for caret blink timing and
        spinner rotation.
    `left_just_pressed()` / `left_just_released()` / `left_held()` are
    convenience queries over the three button Vecs — every widget's
    hit-testing goes through these three methods rather than touching the
    Vecs directly.
  - `NamedKey` — the closed set of non-printable keys agate's text widgets
    understand: arrows, Home/End, Page Up/Down, Backspace, Delete, Enter,
    Tab, Escape. The host's platform key enum (e.g. winit's) gets mapped
    down to this set before being pushed into `UiInput::keys`.

--------------------------------------------------------------------------------
 theme.rs — Color palette, spacing constants, Theme
--------------------------------------------------------------------------------
A flat list of `const Color` values (dark UI palette: backgrounds, controls,
fields, text, syntax-highlight colors) plus point-size and spacing constants,
all consumed by name (`theme::ACCENT`, `theme::PAD`, etc.) throughout
`widgets/` and `text_editor/`.

Notable groups:
  - Surface colors: `BG`, `SURFACE`, `SURFACE_RAISED`, `BORDER`, `SEPARATOR`.
  - Control colors: `CONTROL_BG`, `CONTROL_HOVER`, `CONTROL_ACTIVE`,
    `CONTROL_BORDER`.
  - Field/text-input colors: `FIELD_BG`, `FIELD_BORDER`, `FIELD_FOCUS`,
    `CARET`, `SELECTION_BG`.
  - Semantic colors: `ACCENT`/`ACCENT_HI`/`ACCENT_DIM`, `SUCCESS`, `WARNING`,
    `DANGER`/`DANGER_BG`.
  - Text colors: `TEXT_PRIMARY`, `TEXT_SECONDARY`, `TEXT_DISABLED`,
    `TEXT_ON_ACCENT`, `TEXT_LINK`.
  - Editor-specific colors: `EDITOR_BG`, `GUTTER_BG`, `CURRENT_LINE`,
    `LINE_NUMBER`/`LINE_NUMBER_CUR`, `SCROLLBAR`, and the `SYN_*` JSON
    syntax-highlight colors (`SYN_STRING`, `SYN_NUMBER`, `SYN_KEYWORD`,
    `SYN_KEY` — object keys specifically — `SYN_PUNCT`, `SYN_COMMENT`,
    `SYN_PLAIN`).
  - Chrome colors: `TITLEBAR_BG`, `TOOLBAR_BG`, `SIDEBAR_BG`, `STATUSBAR_BG`.
  - Font sizes (points, pre-scale): `PT_DISPLAY` down to `PT_SMALL`, plus
    `PT_EDITOR`/`PT_EDITOR_LH` (the code editor's font size and line height).
  - Layout constants: `CORNER`/`CORNER_SM`/`CORNER_LG` (rounded-rect radii),
    `PAD`/`PAD_SM`, `ROW_H`, `BTN_H`, `FIELD_H`, `SLIDER_TRACK`/
    `SLIDER_THUMB`, `TOGGLE_W`/`TOGGLE_H`, `CHECK_BOX`.

  - `Theme { scale: f32 }` — the single piece of *state* in this file; every
    other widget takes `&Theme` (or holds a `Theme` copy, since it's `Copy`)
    and calls into it rather than reading the constants directly for
    anything that should respect DPI scaling.
      - `new(scale)` — clamps scale to a sane minimum (`0.25`) so a buggy
        DPI report can't collapse the whole UI to zero size.
      - `px(points)` / `font(points)` — scale a raw point value by
        `self.scale`. (`font` and `px` are currently identical; kept as two
        names so call sites read as "this is a font size" vs "this is a
        layout dimension" even though the math is the same.)
      - `body()` / `small()` — shorthand for `font(PT_BODY)` /
        `font(PT_SMALL)`, since "the standard body text size, scaled" is by
        far the most common query.
  - `Default for Theme` — `Theme::new(1.0)`, i.e. unscaled.

--------------------------------------------------------------------------------
 text_editor/mod.rs — TextEditor state + core accessors
--------------------------------------------------------------------------------
Defines the public `TextEditor` widget type and everything that doesn't
belong in the edit-ops/motion/undo/render split below: construction,
load/save, and the small read-only accessors every caller needs.

  - `Pos { row, col }` — a cursor/selection endpoint, in *character* columns
    (not bytes — see `byte_idx`/`char_len` below for why that distinction
    matters with UTF-8 content). `Ord`-derived so `(a, b)` pairs can be
    sorted into `(start, end)` order with a plain comparison.
  - `EditorGeom` — the pixel geometry computed by the last `build_items`
    call (rect, text origin, line height, char width, visible row count).
    Cached on the editor (`last_geom`) so cursor-motion methods (in
    `motion.rs`) can convert a mouse position to a `Pos`, or know how many
    rows a Page Up/Down should move, *without* the caller re-supplying
    layout info on every keystroke.
  - `TextEditor` fields: the line buffer (`Vec<String>`, one entry per
    line — simplest correct representation for a buffer this size; no rope
    structure needed), `cursor`/`anchor` (selection is "anchor..cursor",
    `None` anchor means no selection), scroll position, `dirty` flag,
    clipboard, undo/redo stacks (see `undo.rs`), the cached `EditorGeom`,
    `tab_width` and `syntax` (toggles JSON syntax highlighting on/off — see
    `render.rs`/`highlight.rs`), and double/triple-click tracking
    (`last_click_at`/`last_click_pos`/`click_count` — see `motion.rs::click`).
  - `empty()` — a fresh single-empty-line buffer, untitled.
  - `load(path)` / `save()` — read/write the buffer as a plain UTF-8 file;
    tabs are expanded to `TAB_STR` (two spaces) on load. `save()` clears
    `dirty` on success.
  - `set_text(text)` — replaces the whole buffer (used by `load`, but also
    a public entry point for "load this string as if it were a file"),
    resets cursor/scroll/undo and the click-tracking state.
  - `text()` — joins the lines back into one `\n`-separated `String`.
  - `file_name()` — the path's file name, or `"untitled"`.
  - `cursor_line_col()` — 1-based `(line, col)` for status-bar display.
  - `line_count()` / `has_selection()`.
  - `selection_range()` / `clear_selection()` / `begin_selection(extend)` —
    the shared selection-state machine every motion/edit method goes
    through: `begin_selection(true)` starts an anchor if one doesn't exist
    yet (used when a Shift-modified motion begins), `begin_selection(false)`
    is a no-op placeholder for the "plain motion" case where the *caller*
    clears the selection itself afterward.
  - `selected_text()` / `delete_selection()` — read or remove whatever's
    between `anchor` and `cursor`; both are share-implementations called
    from `edit_ops.rs` (cut/copy/backspace-over-selection/etc.) and
    `motion.rs` (click-to-collapse-selection).
  - Free functions `char_len`, `byte_idx`, `substr`, `substr_from`,
    `digit_count` — small UTF-8-aware helpers (Rust `String` indexing is
    byte-based, but cursor columns are character-based, so every line edit
    needs to convert) shared by every submodule via `super::`.

--------------------------------------------------------------------------------
 text_editor/edit_ops.rs — Insert/delete/clipboard
--------------------------------------------------------------------------------
All `impl TextEditor` methods that mutate buffer *content* (as opposed to
just moving the cursor):
  - `insert_str(text)` — the one true insertion path; everything else
    (`insert_char`, `newline`, paste) funnels through it. Deletes any active
    selection first, pushes an undo snapshot, expands tabs, and special-
    cases single-line vs. multi-line inserts (a multi-line insert splices
    new lines into the buffer and recomputes the cursor's resulting
    row/col).
  - `insert_char(ch)` — UTF-8-encodes one char and calls `insert_str`.
  - `newline()` — inserts `\n` plus the current line's leading-space indent,
    i.e. auto-indent carries forward.
  - `backspace()` / `delete_forward()` — delete one character (or merge with
    the adjacent line at a buffer edge), or just collapse the selection if
    one exists. Both pop the just-pushed undo snapshot if nothing actually
    changed (e.g. backspace at row 0 col 0), so undo history doesn't fill up
    with no-op entries.
  - `copy()` / `cut()` / `paste()` — operate on the editor's own
    `clipboard: String` field, *not* the OS clipboard — there's no
    cross-application copy/paste here, only within-editor.

--------------------------------------------------------------------------------
 text_editor/motion.rs — Cursor movement, scrolling, mouse mapping
--------------------------------------------------------------------------------
Everything that moves the cursor or viewport without touching buffer
content:
  - `move_left/right/up/down(extend)` — single-character/line motion;
    `extend` is the Shift-held flag, threaded through to `begin_selection`.
  - `move_home(extend)` — toggles between column 0 and the line's first
    non-whitespace column (the common "smart home" editor behavior).
  - `move_end(extend)`, `page(down, extend)` — `page` reads
    `last_geom.visible_rows` to know how far a Page Up/Down should jump.
  - `scroll_by(rows)` — adjusts `scroll_row` directly (mouse wheel).
  - `ensure_visible_default()` / `visible_cols(geom)` / `ensure_visible(...)`
    — the auto-scroll logic that keeps the cursor on-screen; called at the
    end of essentially every motion/edit method. `visible_cols` derives how
    many character columns fit in the current geometry so horizontal
    scrolling (for long lines) works the same way vertical scrolling does.
  - `pos_at(x, y)` — maps a pixel position to a `Pos`, using the cached
    `EditorGeom`; returns `None` if the click was outside the editor's rect.
  - `click(x, y, extend)` — single click moves the caret; agate tracks
    `last_click_at`/`last_click_pos`/`click_count` so a second click within
    400ms and 6px of the first promotes to "select line", and a third click
    promotes to "select all" — the standard double/triple-click convention.
    A Shift-click (`extend == true`) skips the click-counting entirely and
    just extends the existing selection to the new position.
  - `drag_to(x, y)` — extends the selection to the dragged-to position,
    starting a selection anchor on first call if none exists yet.

--------------------------------------------------------------------------------
 text_editor/undo.rs — Undo/redo + select_all/select_line
--------------------------------------------------------------------------------
  - `Snapshot { lines, cursor }` — a full copy of the buffer + cursor at one
    point in time. Whole-buffer snapshots, not diffs — simplest correct
    approach for buffers this size; a 200-entry cap (`push_undo`) bounds the
    memory cost.
  - `push_undo()` — called at the start of every content-mutating operation
    in `edit_ops.rs` (and `delete_selection` in `mod.rs`); clears the redo
    stack on every new edit, per standard undo/redo semantics ("you can't
    redo after making a new edit").
  - `undo()` / `redo()` — pop one stack, push the current state onto the
    other, restore lines+cursor, mark dirty, re-clamp scroll via
    `ensure_visible_default()`.
  - `select_all()` — anchor at (0,0), cursor at the end of the last line.
  - `select_line(row)` — anchor at the start of `row`, cursor at the start
    of the *next* row (so the selection includes the line's trailing
    newline) — or at the end of `row`'s own text if it's the last line.

--------------------------------------------------------------------------------
 text_editor/render.rs — build_items (the draw list)
--------------------------------------------------------------------------------
  - `TextEditor::build_items(rect, theme, font, show_caret, focused, items)`
    — the only method here, and the only place `EditorGeom` gets computed
    and cached. Per call:
      1. Measures a representative glyph (`'0'`) to get a monospace-ish
         `char_w`, computes the gutter width from the line-count's digit
         count, and derives `EditorGeom`.
      2. Clamps `scroll_row` in case the buffer shrank since the last frame
         (e.g. after an undo) and the old scroll position is now out of
         range.
      3. Pushes background/gutter/divider rectangles.
      4. For each visible row: current-line highlight (if no selection),
         selection-range highlight, the line-number label, the line's text
         (via `highlight::highlight_json_line` if `self.syntax`, else
         `highlight::plain_line`), and the caret rectangle if this is the
         cursor's row, caret blink is "on" this frame, and there's no active
         selection.
      5. If the buffer has more rows than fit on screen, appends a
         scrollbar thumb rect sized/positioned from `scroll_row` vs.
         `lines.len()`.
    Local helper `push_rect` builds one `(Area, Item::Shape(Rectangle))`
    pair — kept local to this file since it's only needed here.

--------------------------------------------------------------------------------
 text_editor/highlight.rs — JSON tokenizer + line-to-spans
--------------------------------------------------------------------------------
  - `Kind` / `Tok` — a minimal token classification (string, number,
    true/false/null literal, punctuation, whitespace, `//` comment,
    "other" — i.e. any bare identifier that isn't a JSON literal) and a
    `(start, end, kind)` span over a line's `Vec<char>`.
  - `tokenize(chars)` — a small hand-written scanner; not a general JSON
    parser (it doesn't track nesting or validate structure), just enough to
    color one line correctly in isolation. Handles `\`-escapes inside
    strings, leading-minus negative numbers, and exponent notation.
  - `next_is_colon(toks, from, chars)` — looks ahead past whitespace from a
    string token to see if the next non-whitespace token is `:`, which is
    how `highlight_json_line` decides "this string is an object key" (color
    `SYN_KEY`) vs. "this string is a value" (color `SYN_STRING`) — a
    line-local heuristic, not real JSON parsing, but sufficient since object
    keys are always immediately followed by `:` in valid JSON.
  - `highlight_json_line(line, scroll_col, font, font_px)` — tokenizes the
    line, then walks tokens building `Span`s, skipping/partially-trimming
    tokens that are scrolled off the left edge (`scroll_col`) so horizontal
    scroll doesn't have to re-tokenize from a different start offset.
  - `plain_line(line, scroll_col, font, font_px)` — the `syntax = false`
    fallback: one plain-colored span for the whole visible line, no
    tokenizing at all.

--------------------------------------------------------------------------------
 widgets/mod.rs — Ui struct, frame lifecycle, base primitives
--------------------------------------------------------------------------------
The widget-family `impl Ui` blocks (`buttons.rs`, `sliders.rs`,
`text_input.rs`, `dropdown_scroll.rs`) all extend the `Ui` type defined here.

  - `Rect = [f32; 4]` — `[x, y, w, h]` in window pixels; `in_rect(p, r)` is
    the one hit-test primitive every widget's `is_hovered`/`just_clicked`
    ultimately calls.
  - `WidgetId(u64)` — an opaque per-widget identity, needed anywhere a
    widget has to remember state *across frames* (slider drag origin, text
    field contents/cursor, dropdown open/closed, scroll position) — plain
    immediate-mode draw calls don't have an identity otherwise.
      - `of(label)` — FNV-1a hash of a string label into a `WidgetId`. Callers
        are expected to pass a stable, unique-per-widget label (e.g.
        `"pos_0_{object_index}"`); colliding labels would share state.
  - `WidgetState` — the per-`WidgetId` persisted state: `pressed` (currently
    unused by any widget here, reserved), `slider_drag: Option<(start_value,
    start_mouse_x)>` (shared by `slider`/`drag_float`), `dropdown_open`,
    `scroll_y`, and the text-field trio `text`/`text_cursor`/`text_anchor`
    (`text_anchor` is reserved for a future text-selection feature; not yet
    read anywhere).
  - `PendingTooltip { rect, label }` — sets up a tooltip to be drawn *after*
    everything else this frame (see `finish()`), so a tooltip always paints
    on top regardless of what widget triggered it.
  - `Ui` fields: `theme`, `font` (shared `Arc<Font>` so the same rasterized
    glyphs aren't duplicated per-widget), the accumulated `items` draw list,
    the `state` map, the current frame's `input` snapshot, window size,
    `drag_owner` (which `WidgetId`, if any, owns the mouse-drag this frame —
    ensures only one slider/drag-float responds even if the mouse strays
    over another widget's rect mid-drag), `focused` (which text field has
    keyboard focus), `elapsed` (running clock for caret blink / spinner
    rotation), and the one-frame `tooltip` slot.
  - `new(scale, font)`.
  - `begin_frame(win_w, win_h, input)` — call once per frame before any
    widget calls: snapshots input, clears the draw list and tooltip slot,
    advances `elapsed`, and releases `drag_owner` if the left button was
    released this frame (so a drag can't "stick" past mouse-up).
  - `finish()` — draws the pending tooltip (if any) last, then hands back
    the accumulated `(Area, Item)` list for the frame, via `mem::take`.
  - `draw_tooltip_now(tt)` — sizes a small rounded box above the trigger
    rect (clamped to stay inside the window horizontally) and draws its
    label.
  - `state(id)` / `is_hovered(r)` / `just_clicked(r)` — the three primitives
    every higher-level widget is built from. `just_clicked` is specifically
    "released while still over the rect" (not "pressed over the rect"),
    matching normal button-release semantics (so a press-drag-release
    outside doesn't count as a click).
  - `push_label(...)` / `push_center_label(...)` — append a single-span
    `Item::Text`, either left-aligned at an explicit offset or horizontally
    *and* vertically centered within a rect (the latter measures the text's
    advance width itself, since `ui2d::Text`'s own `Align` enum isn't
    consumed by the layout code per the ui2d docs — see the ui2d overview).
  - `fill(r, color)` / `panel(r, color)` / `card(r)` — flat rect, rounded
    rect, and a "card" (rounded rect + 1px border, the `SURFACE`/`BORDER`
    combo used for the Inspector's Position/Size/Rotation boxes in the
    editor app).
  - `separator(x, y, width)` / `separator_v(...)` — 1px horizontal/vertical
    divider lines.
  - `label`/`label_secondary`/`label_styled`/`label_clipped` — increasingly
    explicit label variants: `label` uses body size/primary color/no clip;
    `label_secondary` is small size/secondary color; `label_styled` exposes
    every parameter; `label_clipped` is body size/primary color with an
    explicit clip rect (the common "text that might overflow a panel" case).

--------------------------------------------------------------------------------
 widgets/draw.rs — Shape-drawing + text-metric primitives
--------------------------------------------------------------------------------
Free functions with no `Ui` dependency — pure "append this shape to this
list" or "measure this text" helpers, shared by every widget file:
  - `push_rect` / `push_rrect` / `push_border` — filled rectangle, filled
    rounded rectangle, and a *stroked* (outline-only) rounded rectangle
    (same shape type, non-zero `stroke` width).
  - `push_ellipse_outline` / `push_ellipse_outline_rotated` — used only by
    `spinner` (sliders.rs) for the static track ring and the rotating arc.
  - `brighten_color` / `dim_color` — multiply RGB channels by a factor,
    clamped at 255 (`dim_color` is just `brighten_color` with a factor < 1;
    kept as a separate name for readability at call sites). Used by
    `button_styled` for the hover/press color shift.
  - `byte_idx_of(s, col)` — same UTF-8 column→byte-offset conversion as
    `text_editor`'s `byte_idx`, duplicated here rather than shared since
    `text_input`'s state model (a single `String` plus a cursor index) is
    independent of `TextEditor`'s line-array model.
  - `text_advance_width(s, up_to_col, font, size_px)` — sums glyph advance
    widths up to a character column; used to position the caret in
    `text_input` and to measure how much text fits before overflowing a
    field's width.
  - `char_index_for_x(text, rel_x, font, size_px)` — the inverse: given a
    pixel offset into a text field, find which character column the click
    landed on (rounds to the nearer glyph boundary rather than always
    flooring, so clicking the right half of a glyph selects *after* it).

--------------------------------------------------------------------------------
 widgets/buttons.rs — Buttons, toggle, checkbox, tabs, list_row, etc.
--------------------------------------------------------------------------------
  - `button` / `button_secondary` / `button_danger` / `button_success` —
    pre-colored convenience wrappers around `button_styled` (accent/
    control/danger/success backgrounds respectively).
  - `button_styled(r, label, bg, fg)` — the shared implementation: hover
    brightens the background by 8%, an active press dims it to 80%, label
    is centered. Returns `just_clicked(r)`.
  - `icon_button(r, icon, tooltip)` — a secondary-styled button that also
    arms a `PendingTooltip` while hovered, if a tooltip string was given.
  - `toggle(r, value, label)` / `checkbox(r, value, label)` — both are
    "stateless" from `Ui`'s point of view (no `WidgetId` needed): they
    compute `Some(!value)` on click and let the *caller* own the persisted
    boolean, rather than agate tracking it. `toggle` draws an
    iOS-style pill+thumb; `checkbox` draws a square with a checkmark glyph
    when true. Both optionally draw a trailing label if non-empty.
  - `badge(r, label, color)` — a small filled pill with centered text, no
    interaction (status indicators, counts).
  - `list_row(r, label, selected)` — a full-width hoverable/selectable row
    (used by the editor app's Navigator for scene-file/object rows):
    selected rows get `ACCENT_DIM` background, hovered-but-unselected rows
    get `CONTROL_HOVER`, otherwise transparent.
  - `disclosure(r, open, label)` — a chevron + label row for collapsible
    sections; returns the *new* desired open state (caller assigns it back),
    same "caller owns the bool" pattern as `toggle`/`checkbox`.
  - `tabs(r, selected, labels)` — divides `r` into `labels.len()` equal
    columns, draws the selected tab with an accent underline, returns
    `Some(index)` if a *different* tab was just clicked (clicking the
    already-selected tab returns `None`).

--------------------------------------------------------------------------------
 widgets/sliders.rs — Sliders, drag_float, progress bars, spinner
--------------------------------------------------------------------------------
  - `slider(id, r, value, range)` — draws a filled track + circular thumb;
    on press over the rect, claims `drag_owner` and records
    `(value, mouse_x)` into `WidgetState::slider_drag`; while it owns the
    drag, converts mouse-x delta into a value delta scaled by the track's
    usable width and the value range, clamped to `range`. Returns
    `Some(new_value)` only when the value actually changed this frame —
    callers are expected to write that back into their own model
    immediately (agate never "owns" the slider's value, only its drag
    gesture).
  - `slider_labeled(id, r, value, range, format)` — reserves space on the
    right of `r` for a formatted value readout (`format` with `{}`
    substituted by `value` to 2 decimal places), then delegates the
    remaining width to `slider`.
  - `drag_float(id, r, value, speed, label)` — a click-and-drag numeric
    field (think Blender/Unity's draggable number widgets): shows an
    optional label on the left and the current value right-aligned;
    dragging converts horizontal mouse delta directly into a value delta
    via `speed` (no on-screen track — purely a drag gesture over the whole
    field rect).
  - `progress_bar(r, progress)` / `progress_bar_labeled(...)` — a filled
    fraction of a rounded track; the labeled variant overlays a centered
    `"NN%"` string.
  - `color_swatch(r, color)` — a filled rounded rect in the given color,
    with a highlighted border on hover; returns whether it was clicked
    (used by the editor app's Inspector to preview — not yet edit — a
    placed object's color).
  - `spinner(cx, cy, radius)` — a static outline ring plus a brighter arc
    that rotates continuously based on `Ui::elapsed` (`360°` per
    `1 / 1.5` seconds) — a busy/loading indicator.

--------------------------------------------------------------------------------
 widgets/text_input.rs — Single-line text field + TextEditor wrapper
--------------------------------------------------------------------------------
  - `text_input(id, r, value, placeholder)` — a single-line editable field:
      - Seeds `WidgetState::text` from `value` the first time this `id` is
        seen (subsequent frames trust the widget's own internal copy, not
        the caller's `value`, until the caller explicitly reacts to a
        `Some(new_string)` return and re-renders with the new value — the
        standard immediate-mode "controlled input" pattern).
      - Click inside focuses the field and places the cursor at the clicked
        character (via `char_index_for_x`); click outside while focused
        un-focuses it.
      - While focused: appends any committed `input.text` characters one at
        a time, *measuring* each candidate insertion's resulting width and
        stopping before it would overflow the field (so the field never
        silently lets text run past its own border); then processes
        `input.keys` for Backspace/Delete/arrow/Home/End.
      - Draws the field background/border (focused border uses
        `FIELD_FOCUS`), the current text or placeholder (placeholder uses
        `TEXT_DISABLED`), and a blinking caret positioned via
        `text_advance_width`.
      - Returns `Some(new_text)` only on a frame where the text actually
        changed; `None` otherwise (including frames where it's simply
        focused and idle).
  - `text_editor(r, editor: &mut TextEditor, focused)` — the bridge between
    `Ui` and the multi-line `TextEditor` widget: calls
    `editor.build_items(...)` to get its draw items appended to `self.items`,
    then forwards this frame's click/drag/scroll straight into the editor's
    own `click`/`drag_to`/`scroll_by` methods. Returns `true` if this call
    is what should claim keyboard focus (i.e. the editor's rect was just
    clicked) — the *host app* (not agate) is expected to track "is the
    editor focused" as part of its own UI state and pass it back in as the
    `focused` parameter next frame, the same pattern `text_input` uses
    `Ui::focused` for internally.

--------------------------------------------------------------------------------
 widgets/dropdown_scroll.rs — Dropdown menu, scroll area, tooltip trigger
--------------------------------------------------------------------------------
  - `dropdown(id, r, selected, options)` — a closed-state button showing the
    selected option + a chevron; clicking toggles `WidgetState::dropdown_open`.
    While open, draws a list of option rows below the button (selected row
    tinted `ACCENT_DIM`, hovered rows `CONTROL_HOVER`); clicking an option
    returns `Some(index)` and closes the list; clicking anywhere outside
    both the button and the open list also closes it (without returning a
    value).
  - `scroll_area(id, r, content_height)` — *not* a clipping container (ui2d's
    `Area::bounds` clipping is the caller's job) — just the scroll-offset
    bookkeeping: draws the area's background, accumulates wheel input into
    `WidgetState::scroll_y` (only while hovered), clamps to
    `[0, content_height - r.height]`, and returns `(r, scroll_y)` for the
    caller to subtract from its own content's draw positions.
  - `end_scroll_area(id, r, content_height)` — draws the scrollbar thumb for
    the `scroll_area` started above (no-ops if everything fits, i.e.
    `content_height <= r.height`); separate call so the caller can draw
    content *between* `scroll_area` and `end_scroll_area`.
  - `tooltip(trigger_rect, label)` — the manual equivalent of what
    `icon_button` does automatically: arms a `PendingTooltip` if
    `trigger_rect` is currently hovered, for any custom widget that wants
    hover-tooltip behavior without being a button.

================================================================================
 ARCHITECTURE SUMMARY
================================================================================

Two independent widgets share one foundation:

  1. `theme.rs` + `input.rs` — the foundation. `Theme` answers "how big/what
     color should this be drawn", scaled for DPI; `UiInput` answers "what
     did the user do this frame". Neither depends on the other or on `Ui`.

  2. `text_editor/` — a self-contained multi-line buffer widget. It owns its
     *own* selection/scroll/undo state directly (not through `Ui::state`,
     since there's only ever one `TextEditor` instance per editor tab rather
     than many identical widgets needing per-`WidgetId` state). `Ui` only
     touches it through one bridge method, `widgets::text_input::text_editor`.

  3. `widgets/` — the general immediate-mode widget set. `mod.rs` is the
     foundation (frame lifecycle + the four-or-so primitive draw helpers
     every other widget file calls into); `draw.rs` is pure helper functions
     with no `Ui` dependency at all; the other four files
     (`buttons`/`sliders`/`text_input`/`dropdown_scroll`) are `impl Ui`
     blocks grouped by widget family, all able to freely access `Ui`'s
     private fields and `WidgetState`'s private fields because Rust treats
     "private to the module that defined it" as "visible to that module and
     all its descendants" — `widgets::buttons` etc. are descendants of
     `widgets`, where `Ui`/`WidgetState` are defined.

  Call pattern for a host app, every frame:
     `ui.begin_frame(win_w, win_h, &input)`
     → any number of widget calls (each both draws and returns interaction)
     → `let items = ui.finish();`
     → hand `items` to `space_soup::ui2d::Overlay::set_items` (or a
       `WorldPanel`) for actual GPU rendering.
